# 简单GTO V1 实施计划

- 日期: 2026-04-23
- 对应 PRD: `docs/specs/2026-04-23-simple-gto-v1-prd.md`
- 状态: 待执行

## 1. 实施目标

基于已确权 PRD，按最小可用、可验证、可迭代的顺序实现 V1。

实施优先级:

1. 先保证牌局规则正确
2. 再保证实战桌交互顺手
3. 再接入 bot 行为与赛后总结
4. 最后补策略面板与轻量 Study

## 2. 开发前约束

### 2.1 Git / Worktree

PRD 要求后续开发在独立 worktree 中进行。

当前工作区现状:

- `D:\CodeProjects\简单GTO` 还不是 git 仓库

因此，正式编码前需要先完成:

1. 初始化 git 仓库
2. 建立主开发分支
3. 创建独立 worktree 作为实施副本

说明:

- 在当前阶段只写计划，不直接开始污染主目录业务代码
- 真正开始实现时，再按 worktree 流程执行

### 2.2 首版技术原则

- 优先单体架构，不拆服务
- 优先本地可跑通，不先做部署体系
- 优先移动端竖屏体验
- 优先规则正确与 bot 真实感
- 不提前为多用户、多人联机、云同步做设计

## 3. 推荐技术方案

## 3.1 前端

建议:

- Next.js 或 Vite + React
- TypeScript
- Tailwind 或等价轻量样式系统

推荐方向:

- React + TypeScript
- 优先组件化移动端 UI
- 竖屏桌面布局以 CSS 变量和固定区域锚点驱动

## 3.2 状态管理

建议:

- 牌局状态机独立于 UI
- 前端只消费当前可渲染状态和合法动作

推荐:

- 轻量全局状态管理
- 明确分层:
  - `game-engine`
  - `bot-engine`
  - `ui-shell`
  - `post-session-review`

## 3.3 数据持久化

V1 最小化:

- 不做账号数据库
- 不做历史 session 档案
- 仅保留当前 session 内存状态

如需少量本地恢复:

- 仅允许保存“继续上次牌桌”所需的当前 session 快照
- 不扩展为历史档案系统

## 4. 模块拆分

### 模块 A: App 外壳与首页

目标:

- 打开即进入极简首页
- 支持 `开始实战`
- 支持 `继续上次牌桌`
- 手机竖屏适配

交付:

- App 路由骨架
- 首页 UI
- PWA 基础壳

### 模块 B: 牌局引擎

目标:

- 具备完整 NLHE 现金局状态机

包含:

- 发牌
- 按钮轮换
- SB/BB 扣盲
- 街道推进
- 合法动作计算
- 多人弃牌提前结束
- 摊牌比较
- 抽水
- 补码
- session 计时

这是 V1 最优先模块。

### 模块 C: 牌桌 UI

目标:

- 在手机竖屏下稳定显示 8 人桌

包含:

- 顶部桌况信息
- 中部牌桌与 8 席位
- 用户手牌锚点
- 主操作区
- 底部 `策略 / 选项`
- 金额 + BB 双显示

### 模块 D: 下注器与预选动作

目标:

- 做出接近线上现金局的操作手感

包含:

- `Fold / Check-Call / Bet-Raise / All-in`
- 快捷下注尺寸
- 手动微调
- 预选动作
- 行动倒计时与 time bank
- 超时默认动作

### 模块 E: Bot 引擎

目标:

- 先做出“像真人强手桌”的基础行为

拆分:

- 顶级强手池
- 偏离型风格池
- 桌池生成器
- 行为延迟器
- 动作选择器

V1 不追求复杂 AI，自上而下采用:

- 规则化范围
- 上下文决策模板
- 风格参数偏移

### 模块 F: 统一 Preflop 范围表

目标:

- 只做一个底部上拉面板

包含:

- 座位旁扑克牌图标入口
- 底部 `策略` 入口
- 统一面板
- 不遮挡手牌与操作区
- 动作颜色固定

### 模块 G: 暂停提示与赛后页

目标:

- 有明显错误时轻量暂停
- 整场结束后输出赛后页

包含:

- 街道标签
- 一句话中文错误提示
- bot 个性揭示
- VPIP / PFR / 本场输赢
- 用户本场输赢

## 5. 批次计划

### 批次 1: 项目骨架

交付:

- 前端项目初始化
- PWA 基础配置
- 首页
- 路由与目录结构
- 基础设计 token

验收:

- 手机浏览器可打开
- 首页可进入空白牌桌页

### 批次 2: 牌局状态机最小闭环

交付:

- 发牌
- 扣盲
- 行动顺序
- Fold / Check / Call / Bet / Raise / All-in
- 街道推进
- 摊牌判断

验收:

- 可完整跑通一手牌
- 可在任意街道自然结束

### 批次 3: 8 人桌 UI 与下注器

交付:

- 竖屏 8 人桌布局
- 用户手牌锚点
- 主操作区
- 快捷下注按钮
- 双显示

验收:

- 安卓手机竖屏可打牌
- 不发生核心区域遮挡

### 批次 4: 现金局规则完善

交付:

- 抽水 5% 封顶 3BB
- 补码规则
- action timer
- time bank
- 预选动作
- session 60 分钟结束逻辑

验收:

- 规则与 PRD 对齐

### 批次 5: Bot 桌池

交付:

- 3 顶级强手
- 4 偏离型 bot
- 同风格最多 2 人规则
- 基础行动延迟

验收:

- 一整桌可自然运行
- bot 行为不明显脚本化

### 批次 6: 策略面板

交付:

- 统一底部范围表
- 双入口
- Preflop 动作展示

验收:

- 展开时不遮挡手牌
- 可边打边看

### 批次 7: 暂停与赛后页

交付:

- 一句话错误提示
- 整场结束赛后页
- bot 个性与数据揭示

验收:

- 整场跑通后有清晰总结

## 6. 风险点

### 6.1 最高风险

- 8 人竖屏布局过密
- bot 真实感不够
- 牌局状态机边界遗漏

### 6.2 中风险

- 范围表和手牌区互相挤压
- 快捷下注尺寸不顺手
- 赛后页信息层级混乱

### 6.3 应对策略

- 先做状态机，再做 UI 美化
- 先做最小 bot，再逐步加风格差异
- 先做 session 内回看，不做长期档案

## 7. 建议执行顺序

建议按如下顺序实际开工:

1. 初始化 git 仓库并建立 worktree 开发副本
2. 搭前端基础壳
3. 实现最小牌局状态机
4. 接 8 人桌 UI
5. 接下注器与计时
6. 接 bot 桌池
7. 接范围表
8. 接赛后页

## 8. 完成定义

当以下条件全部满足时，V1 视为可验收:

- 安卓手机上可像 App 一样打开
- 首页一键开始实战
- 8 人桌完整跑通
- 规则符合 PRD
- bot 桌池像真实现金桌
- 范围表能边打边看
- 赛后页能看懂桌上人物和本场结果
